Motivation

You've got to be joking right?

Of course! But maybe...

 

Louis C.K. did a great routine that I always think about when I talk about Descartes. He talks about a class of thoughts he has that he doesn't believe - truly outlandish horrid things - but that he can't help but feel might make sense. It's a category of thoughts he calls "Of course! But maybe..."


Of course, writing CSS in JavaScript is a terrible idea! Of course! But maybe...

Are you a wizard?


Of course, you should use CSS to do your styling whenever possible. If I want to simply vertically align something like a div, this makes absolutely perfect sense.

.verticalAlign {
  position: relative;
  top: 50%;
  transform: translateY(-50%);
}

...right? Of course! Of course this makes sense!

But maybe...that's just stupidly unreadable

Maybe writing CSS can feel like you're invoking magical spells. Maybe if I really wanted to make a div element vertically align, I should be able to do so with some basic math. Something sensible might use references to the DOM like this.

.verticalAlign {
  margin-top: (parent.height - this.height) / 2 + "px";
}

But of course not! DOM access and JavaScript in CSS is crazy! Of course...

Cascading Cluster Spasms


You've also probably had some nightmares about how cascading works. Why is everything global? Why does specificity denote priority? If you had to write plain CSS lately, you know it gets really messy really quickly with all the namespacing and repitition.

html {
  margin: 0;
  padding: 0;
}
html body {
  margin: 0;
  padding: 0;
}
html body section {
  max-width: 800px;
}

But wait! We have things like LESS and SASS, you say. Nesting rules together lets you get rid of worrying about specificity and even gives you pretty things like variables and mixins. You can turn the CSS we had earlier and use a preprocessor instead.

html {
  margin: 0;
  padding: 0;
  body {
    margin: 0;
    padding: 0;
    section {
      max-width: 800px;
    }
  }
}

All you need to do is learn a new language and install the preprocessor and limit yourself to something like variables and some functions and stuff. Easy, right?

But maybe...we already have JavaScript?

We could represent what you want in LESS or SASS in a JavaScript object literal and apply the styles yourself. You would also get all the nice built-in functionality of JavaScript with variables, loops, conditionals, functions, and you wouldn't need to use another technology pattern completely, maybe?

// LESS or SASS

html {
  margin: 0;
  padding: 0;
  body {
    margin: 0;
    padding: 0;
    section {
      max-width: 800px;
    }
  }
}
var styles = {
  "html": {
    "margin": 0,
    "padding": 0,
    "body": {
      "margin": 0,
      "padding": 0,
      "section": {
        "max-width": "800px"
      }
    }
  }
}

Of course not! Separation of responsibilities! I wouldn't dream of doing that in production! Of course...

index.html 
<!doctype html>
<html>
  <head>
    <script type="text/javascript" src="descartes.js"></script>
  </head>
  <body>
    <h1>Hello World!</h1>
  <script type="text/javascript" src="styles.js"></script>
  </body>
</html>
 
styles.js 
var rand_angle = function() {
  return Math.round(Math.random() * (180) - 90);
};
var rand_rgba = function() {
  return "rgba("+[255,255,255].map(function(x) {
    return Math.round(Math.random() * x);
  }).join()+", 1)"
};

new Descartes({ // Just put descartes.js in the <head> tag
  "html": {
    "body": { // Nest selectors just like in Sass and Less
      "font-family": "Arial", // Set properties
      "font-size": 10 + 6, // = 16, converts to px
      "$(body).keyup": { // Bind events
        "background": function() { // To certain properties
          return 'linear-gradient('
            + rand_angle().toString() + 'deg,'
            + rand_rgba() + ','
            + rand_rgba() + ')'
        } // ...and use functions to generate property values!
      }
    }
  }
})
